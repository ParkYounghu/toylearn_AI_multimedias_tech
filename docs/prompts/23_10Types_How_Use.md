## 가상 문제 상황
1. 가상 문제 상황 (Scenario Challenge)
페르소나 (Persona):
직무: 헬스케어 스타트업 '웰니스 핏(Wellness Fit)'의 주니어 PM (Product Manager)
성향: 아이디어는 많지만, 이를 개발자나 디자이너가 이해할 수 있는 기술적 언어로 정리하는 데 어려움을 겪고 있음.
배경 (Background):
당신은 개인 맞춤형 영양 관리 웹 서비스인 '뉴트리 가이드(Nutri-Guide)'의 MVP(최소 기능 제품) 웹 프로토타입을 기획 중입니다.
일주일 뒤, 개발팀 및 디자인팀과 함께 '프로토타입 제작 킥오프 미팅'이 예정되어 있습니다.
현재는 "사용자가 식단을 입력하면 칼로리를 계산해준다", "깔끔한 대시보드가 있었으면 좋겠다" 정도의 추상적인 아이디어만 나열된 상태입니다.
핵심 문제 (Problem):
개발 리더로부터 **"기능 명세가 너무 모호해서 DB 설계나 프론트엔드 구조를 잡을 수 없다"**는 피드백을 받았습니다.
구체적인 필수 기능 리스트(Feature List), 사용자 흐름(User Flow), 데이터 요구사항이 정의되지 않으면 프로젝트 착수가 무기한 연기될 위기입니다.
해결 과제 (Task):
추상적인 아이디어를 개발자와 디자이너가 즉시 작업할 수 있는 수준의 **'웹 프로토타입 제작을 위한 상세 요구사항 리스트'**로 변환해야 합니다.
기능적 요구사항(회원가입, 식단 입력, 리포트 출력 등)과 비기능적 요구사항(보안, 로딩 속도 등), 그리고 필요한 페이지 구조를 빠짐없이 도출해내야 합니다.

2. 훈련 초점 (Training Focus)
이 문제는 [구조적 분석 및 세분화 (Structural Breakdown)] 능력을 평가.
단순히 "아이디어를 내줘"가 아니라, 모호한 요청을 **체계적인 카테고리(Front-end, Back-end, UX/UI 등)**로 나누어 구체화하는 프롬프팅 스킬이 필요.
## 문제 해결 프롬프트 

너는 헬스케어 스타트업 '웰니스 핏(Wellness Fit)'의 **주니어 PM** 역할을 수행한다. 현재 개발 리더로부터 **"기능 명세가 모호하다"**는 피드백을 받은 상황을 해결해야 한다.

**[상황 목표]**
일주일 뒤 킥오프 미팅에서 프로젝트 착수를 확정 짓기 위해, 추상적인 아이디어를 **개발/디자인팀이 즉시 작업 가능한 수준의 구체적이고 체계적인 요구사항 명세서**로 변환해야 한다.

**[핵심 요구사항 및 가중치]**
* **W1 (가중치 40%): 구조적 명확성:** 기능적 요구사항(FR), 비기능적 요구사항(NFR), 데이터 요구사항을 **기술적 카테고리**로 철저히 분리할 것.
* **W2 (가중치 30%): 기술 스택 명시:** 프론트엔드와 백엔드에 사용될 **주요 기술 스택**을 PM이 제안하는 형식으로 명시할 것.
* **W3 (가중치 30%): UX 시각화:** 각 페이지 구조에 **와이어프레임 레벨**의 구체적인 UI/UX 흐름을 명시할 것.

**1. 분석 및 전략 목표 (CoT, 인스턴스 기반 필수):**
* **Chain of Thought (6):** '추상 아이디어(식단, 대시보드) 분석' → '기술 스택(Front/Back) 정의' → '기능/비기능/데이터 요구사항 분리 및 명확화' → '최종 명세서 산출' 단계를 밟을 것.
* **인스턴스 기반 (9):** 상세 기능 리스트(FR) 도출 시, **경쟁 서비스인 'MyFitnessPal'의 검색 기반 식단 입력 방식**과 **'MyPlate'의 시각화 대시보드 구조**를 참고하여 요구사항의 전문성을 높일 것.

**2. 출력 형식 및 예시 기반 통제 (기술 명세서 구조):**
답변은 반드시 하나의 **Markdown 코드 블록**을 사용하며, 다음 세 가지 섹션으로 구성된 **기술 명세서** 형식으로 제시한다.

* **섹션 A: 프로젝트 개요 및 기술 스택 제안:** 목표 및 예상 기술 스택 (Front-end, Back-end)을 간략히 표로 정리.
* **섹션 B: MVP 기능 및 데이터 명세 (FR):** **ID, 기능 카테고리, 상세 요구사항 (Front-end/UX), 데이터 요구사항 (Back-end/DB)**을 포함하는 표.
* **섹션 C: 비기능적 요구사항 (NFR):** **ID, 카테고리, 상세 요구사항 (인프라/보안), 측정 기준 (KPI)**을 포함하는 표.

## 섹션 B: MVP 기능 및 데이터 명세 (FR) (W1: 구조적 명확성, W3: UX 시각화)

| ID | 기능 카테고리 | 상세 요구사항 (Front-end/UX 와이어프레임 레벨) | 데이터 요구사항 (Back-end/DB 필드) |
| :--- | :--- | :--- | :--- |
| **FR-01** | **사용자 인증** | **로그인/회원가입:** 이메일/비밀번호 입력 폼 및 소셜 로그인 버튼 (Google, Apple). **와이어프레임:** 1. 단일 화면에서 로그인/회원가입 탭으로 전환 → 2. 성공 시 대시보드 페이지 이동. | `User(user_id: PK, email: UNIQUE, password_hash, created_at)` |
| **FR-02** | **식단 검색/입력** | **식단 검색 및 입력 (MyFitnessPal 방식):** 상단 검색창 (실시간 자동 완성), 음식 선택 후 **섭취량 (단위, 수량)** 입력 모달 팝업. **와이어프레임:** 1. 식단 기록 Floating Action Button → 2. 음식 검색 페이지 → 3. 검색 결과 목록 (자동 완성) → 4. 항목 선택 시 '섭취량/단위' 입력 **모달** → 5. 저장 후 식단 목록 갱신. | `Meal(meal_id: PK, user_id: FK, type: {아침, 점심, 저녁, 간식}, food_name, quantity, unit, kcal: Int, protein: Float, fat: Float, carb: Float, created_at: Date)` |
| **FR-03** | **대시보드 시각화** | **일일 영양 목표 대비 시각화 (MyPlate 방식):** 일일 총 칼로리/영양소 **목표 대비 섭취량** 도넛 차트 또는 원형 차트. **와이어프레임:** 1. 일일 목표 (텍스트) → 2. 핵심 영양소 (단백질, 지방, 탄수화물) 달성률 **도넛 차트** → 3. 식사별 기록 목록. | `Goal(user_id: FK, target_kcal: Int, target_protein: Float, target_fat: Float, target_carb: Float)` DB는 FR-02의 데이터를 일자별로 집계(Aggregate)하여 API로 제공해야 함. |
| **FR-04** | **데이터 수정/삭제** | **기록 항목 편집:** 기록된 식단 항목을 **좌우 스와이프**하여 '삭제' 또는 '편집' 옵션 노출. 편집 시 FR-02와 동일한 입력 모달 재활용. | DB의 `Meal` 테이블 데이터에 대한 **UPDATE/DELETE** API 제공. |

## 섹션 C: 비기능적 요구사항 (NFR) (W1: 구조적 명확성)

| ID | 카테고리 | 상세 요구사항 (인프라/보안/성능) | 측정 기준 (KPI) |
| :--- | :--- | :--- | :--- |
| **NFR-01** | **성능** | 식단 검색 시 **타이핑 후 1초 이내**에 검색 결과 목록이 표시되어야 하며, 대시보드 로딩 시간은 2초를 넘기지 않아야 함. | 평균 API 응답 시간: **\< 1000ms** (식단 검색 API), 대시보드 완전 로딩 시간: **\< 2000ms** |
| **NFR-02** | **보안** | 모든 사용자 인증 및 민감 데이터(비밀번호, 식단 기록) 전송 시 **SSL/TLS 암호화**를 필수 적용해야 함. 비밀번호는 최소 10자 이상, 해시 저장(Bcrypt 권장). | **OWASP Top 10** 주요 항목 (특히 A2, A7)에 대한 취약점 점검 통과. |
| **NFR-03** | **확장성** | 일일 최대 **5,000명**의 활성 사용자를 안정적으로 처리할 수 있도록 서버 리소스(Auto Scaling)를 설정해야 함. 장기적으로는 1만 명을 목표로 DB 인덱싱을 최적화해야 함. | 서버 부하율 **피크타임 평균 60% 이하** 유지. |
| **NFR-04** | **접근성** | 모바일 앱의 **글자 크기 조정 기능**을 지원하고, 주요 인터랙티브 요소는 **최소 44x44pt** 터치 영역을 확보해야 함. 색상 대비 기준(WCAG AA)을 준수. | A/B 테스트 시, 노년층 사용자의 **식단 입력 성공률 90% 이상**. |
